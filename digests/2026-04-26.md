## 3. Video AI / Multimodal

**Gemini 3 Pro >1 FPS sampling** is the headline change for this app. Soccer shots and passes unfold in 200–500 ms; the previous ≤1 FPS sampling could miss events entirely. Combined with the 100 MB upload limit increase, full-session recordings can now be sent whole rather than chunked, simplifying `server/services/videoAnalyzer.js` significantly. Use `media_resolution` parameter to balance token cost vs. visual fidelity for long sessions. ([video understanding docs](https://ai.google.dev/gemini-api/docs/video-understanding))

**Gemini 3.1 Flash-Lite** is a strong candidate for a two-pass pipeline: run Flash-Lite first for cheap clip filtering ("does this 30-second window contain a shot?"), then send confirmed-interesting segments to Gemini 3 Pro for structured event extraction. This reduces Pro token spend significantly at scale.

**SoccerNet 2026 Challenges** (sixth edition) closed submissions April 24 ([soccer-net.org](https://www.soccer-net.org/challenges/2026)). New this year: **Player-Centric Ball Action Spotting** — identifying not just *what* and *when* but *who* performed the action. This is the open-research equivalent of the app's shot/pass detection feature. Post-competition leaderboards (published in May) will be a useful accuracy benchmark for your own pipeline.

A [Nature Scientific Reports study](https://www.nature.com/articles/s41598-025-93159-2) on youth football achieved 98.3% accuracy classifying jogging/sprinting/passing/shooting/jumping using CNN + BiLSTM on skeleton data — the same age group as this app's users. The architecture is a credible reference for a small distilled model for in-browser inference via WebGPU.

**MediaPipe Pose** provides 33 3D landmarks at real-time on-device speeds across web/iOS/Android ([IT-JIM writeup](https://www.it-jim.com/blog/mediapipe-for-sports-apps/)). A hybrid pipeline (MediaPipe for player skeletons → Gemini for event semantics) would reduce cloud costs and provide privacy-friendly local processing — useful for youth data compliance. Known limitation: accuracy degrades for distant players and when limbs point toward the camera.

## 5. Browser / PWA Capabilities

### WebGPU (Chrome 147–148, April 2026)
New compute shader built-ins: `global_invocation_index` and `workgroup_index` simplify linear-array indexing ([What's New in WebGPU 147-148](https://developer.chrome.com/blog/new-in-webgpu-147-148?hl=en)). WGSL gains limited pointer support. All major desktop browsers now ship WebGPU; Firefox behind a flag, Safari partial in Tech Preview. **Relevance:** WebGPU is now viable for running MediaPipe Pose or MoveNet via TensorFlow.js on-device — important for iPad sideline use cases and privacy-safe player tracking without cloud round-trips.

### OPFS
2026 consensus is that OPFS (Origin Private File System) is preferred over Cache API for multi-GB assets ([Definitive Guide to Local-First AI, SitePoint](https://www.sitepoint.com/definitive-guide-local-first-ai-2026/)) because it supports streaming reads without loading entire files into memory. **For dual-mode recording in `src/utils/videoRecorder.js`:** stage recorded sessions to OPFS before the tus upload rather than holding them in memory or IndexedDB blobs. Prevents tab crashes for full-session recordings at 1080p.

### MediaRecorder / WebCodecs
HEVC (H.265) platform encoding now supported in MediaRecorder and WebCodecs (Chromium M130+) ([Chromium MediaRecorder README](https://chromium.googlesource.com/chromium/src/+/HEAD/third_party/blink/renderer/modules/mediarecorder/README.md)). HEVC at the same quality as H.264 is roughly half the file size — direct reduction in Gemini upload latency and storage costs. Codec fragmentation across browsers (WebM/VP8/Opus) is largely resolved in 2026. **Recommended pattern:** prefer WebCodecs (hardware-accelerated) for capture + MediaRecorder for the container, and reserve FFmpeg.wasm for server-side trim/mux only.

### PWA / Service Workers
- Workbox 7 (used by `vite-plugin-pwa` v1.2.0) has native Vite/webpack/Next.js integrations. Current `vite.config.js` `NetworkFirst` strategy for `/api/` is correct. ([Service Workers in PWA 2026](https://jsmanifest.com/service-workers-pwa-guide))
- iOS Safari PWA limitations remain the top source of production bugs in 2026: limited storage quotas, partial push support, recording permission behaviour differs from Chrome. ([PWA iOS Limitations 2026](https://www.magicbell.com/blog/pwa-ios-limitations-safari-support-complete-guide)) **Test recording and tus uploads on iOS Safari explicitly** — especially on the `LiveSessionMode` component.
- Custom install prompts are preferred over default browser banners; consider adding a custom prompt for coach users who haven't installed the PWA.

## 6. Auth / Data Layer

### better-sqlite3 12.9.0 + SQLite 3.53.0
**SQLite 3.53.0** (April 9) is bundled in better-sqlite3 12.9.0 ([release notes](https://sqlite.org/releaselog/3_53_0.html), [Simon Willison's writeup](https://simonwillison.net/2026/Apr/11/sqlite/)). Key additions:
- ALTER TABLE can now **add/drop NOT NULL and CHECK constraints** — long-standing pain point for incremental schema evolution in `server/db.js`.
- EXISTS-to-JOIN optimiser improvement — relevant for session/match join queries in `server/routes/coachDashboard.js`.
- Float-to-text rounding bumped 15 → 17 significant digits (default) — verify metric formatting in `src/utils/stats.js` is not display-sensitive.
- REINDEX EXPRESSIONS statement for rebuilding stale expression indexes.
- `3.52.0` was withdrawn due to compatibility issues; `3.53.0` supersedes it — no gap.

### Drizzle ORM
**0.45.2** stable / **1.0.0-beta.2** in flight ([latest releases](https://orm.drizzle.team/docs/latest-releases)). Native `node:sqlite` auto-detection at runtime has been added. If you ever want a typed query layer over better-sqlite3, Drizzle is the right choice — the 1.0 beta signals API stabilisation. No breaking changes to existing raw better-sqlite3 queries in `server/db.js`.

### JWT alternatives
PASETO is gaining traction for internal API auth (eliminates algorithm-confusion class of vulnerabilities), while JWT remains standard for external IdP interop (the app's Google OAuth flow). ([MojoAuth analysis](https://mojoauth.com/blog/jwt-vs-paseto-vs-branca-the-future-of-secure-tokens-in-2026)) No immediate need to migrate, but the `server/auth.js` token generation is worth hardening with explicit algorithm pinning (`{ algorithms: ['HS256'] }`) in `jsonwebtoken` verify calls to close the algorithm-confusion vector.

### WebAuthn / Passkeys
PRF (Pseudo-Random Function) extension is now mature: Firefox 148+ full support, Chrome 147 ships PRF-on-create by default ([Corbado writeup](https://www.corbado.com/blog/passkeys-prf-webauthn)). FIDO Alliance is finalising cross-provider passkey portability standards. **For a youth app where parents manage accounts**, passkeys + email-magic-link fallback is a credible auth upgrade over username/password — avoids the shared-password problem common in family accounts.

### react-webcam
See §2. The library is dormant. Evaluate `react-record-webcam` 2.0.1 (actively maintained, records natively) as a replacement in `src/components/LiveSessionMode.jsx` and `src/components/CameraSetup.jsx`.
